# Compte rendu

Aujourd’hui, nous avons principalement travaillé sur la **mise en place du socle technique du multijoueur** dans le projet HAVEN.

L’objectif de la journée était de préparer une architecture solide avant d’intégrer un vrai système réseau (Steam et NGO plus tard). Nous avons donc commencé par structurer le projet afin de bien séparer les différentes responsabilités entre les systèmes du jeu.

Nous avons d’abord finalisé l’architecture principale du projet en séparant clairement les différentes couches :
- **Core** pour les systèmes génériques (EventBus, logging, gestion des scènes, configuration).
- **Network** pour tout ce qui concerne la session et l’état réseau.
- **App** pour le bootstrap et l’initialisation globale du jeu.
- **UI** pour les vues et les contrôleurs utilisés par l’interface.

Cette organisation permet de garder un projet plus propre et plus maintenable, tout en évitant les dépendances circulaires entre les modules.

Ensuite, nous avons implémenté un **NetworkService**, qui sert de runtime réseau central. Ce service permet de gérer l’état du réseau et les différents modes de fonctionnement du jeu (Local, Host, Client). Il publie également des événements afin que les autres systèmes du projet puissent réagir aux changements d’état réseau.

Nous avons ensuite connecté ce système avec le **SessionService**, qui gère le cycle de vie d’une session de jeu (menu, lobby, mission). Cela permet maintenant à la session de piloter le mode réseau selon le type de partie lancée.

Nous avons également ajouté les différents **flows de session** pour préparer le multijoueur :
- création d’un lobby local,
- création d’un lobby en tant qu’hôte,
- rejoindre une session en tant que client.

Ces flows sont accessibles depuis le **Main Menu**, ce qui permet déjà de tester les transitions entre les différents états du jeu.

Une partie importante du travail a également consisté à **corriger la gestion du mode réseau lors des transitions de scènes**. Lors du retour d’une mission vers le lobby ou vers le menu principal, le système préservait ou arrêtait maintenant correctement le runtime réseau selon le contexte.

Enfin, nous avons réalisé plusieurs **tests complets en build** afin de vérifier que :
- le bootstrap du projet fonctionne correctement,
- les transitions de scènes sont stables,
- les états de session et les états réseau restent cohérents,
- les flows Local, Host et Client fonctionnent sans erreurs.

Ces tests permettent de valider que la base technique du projet est maintenant suffisamment solide pour commencer l’intégration d’un **vrai système multijoueur** lors des prochaines étapes.
